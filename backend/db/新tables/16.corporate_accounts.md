# corporate_accounts

## 概要
企業アカウント（`account_list.parent_account_type = '企業アカウント'`）向けの派生情報を保持するテーブル。  
`account_list` を正規化のまま維持し、ここでは「中ジャンル（second）」「第三目的（third）」「アカウント頻出ハッシュタグ」を格納する。

## テーブル定義

| 列名 | データ型 | NULL | デフォルト | インデックス | 説明 |
|-----|---------|------|----------|------------|------|
| account_id | INT | NO | - | PRIMARY, FK | `account_list.id` を参照 |
| second_account_type | VARCHAR(50) | YES | NULL | idx_second_account_type | 中ジャンル（'採用'または'集客'） |
| third_account_type | VARCHAR(255) | YES | NULL | idx_third_account_type | 両方含む場合のもう一方（なければ NULL） |
| account_hashtags | TEXT | YES | NULL | - | アカウントの頻出ハッシュタグ（頻出順, カンマ区切り） |
| created_at | DATETIME | NO | CURRENT_TIMESTAMP | - | 作成日時 |
| updated_at | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | - | 更新日時 |

## インデックス

| インデックス名 | 列名 | 種類 | 説明 |
|--------------|------|------|------|
| PRIMARY | account_id | PRIMARY | 主キー |
| idx_second_account_type | second_account_type | INDEX | 中ジャンルでの絞り込み |
| idx_third_account_type | third_account_type | INDEX | 第三目的での絞り込み |

## 関連テーブル

| このテーブルの列名 | テーブル名.列名 | 関連タイプ | 説明 |
|-----------|-----------|-----------|------|
| account_id | account_list.id | 多対一 | 元アカウント情報 |

## データ生成/更新
1. Pub/Subトリガ同期（スプレッドシート由来）
   - フロー: `sync_account_list` 完了 → Pub/Sub `sync-corporate-accounts` → `sync_corporate_accounts_from_sheet`
   - シート: 「アカウント作業用シート」
   - 列マッピング（B:N取得時の相対位置）
     - B列: account_url（キー）
     - H列: second_account_type
     - I列: third_account_type
     - J列: account_hashtags
   - `account_url` をキーに `account_list.id` を取得し、`corporate_accounts` を UPSERT。
   - 企業判定: `account_list.parent_account_type = '企業アカウント'` かつ `favorite_user_is_alive = 1` のみ対象。

2. バッチ再集計（任意）
   - account_hashtags を動画由来に再集計: `video_hashtags × video_master` で username 単位に頻度集計し、頻出順で `GROUP_CONCAT`。
   - 更新頻度: 日次や集計完了後に `INSERT ... ON DUPLICATE KEY UPDATE` で再計算。

注意:
- ハッシュタグ連結長は `group_concat_max_len` を適宜拡張。
- 期間限定集計が必要な場合は `video_master.post_time` による WHERE 条件を追加。

## 備考
- 正規化を維持するため、`account_name`, `account_url`, `account_type` はこのテーブルに保持しない。表示/APIはビュー `v_corporate_accounts` で JOIN して提供。
- 将来的に third_account_type の定義を拡張（例: 追加の目的分類）する場合は CASE ロジックを差し替え可能。
